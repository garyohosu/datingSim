## Q6. stressパラメータの効果（Sec.8.2 / 8.3）

**疑問**: 高値時の影響・低減手段が未定義。

**回答**:
- 0〜29: 問題なし
- 30〜59: 一部選択肢の効果が弱くなる
- 60〜79: 好感度上昇が低下
- 80〜99: 体調不良イベント、失敗率上昇

低減手段: 休養系イベント、穏やかな日常、特定ヒロインとの安らぎイベント、休日、選択肢による回復。

## R1. 分岐エピソードと「1日 = 1ターン」の対応（Sec.7.1 / 7.2 / 11.2）

**回答**:
- `day-002a` / `day-002b` は **どちらも同じ Day 2 の分岐差分**
- `1日 = 1ターン` は維持し、分岐してもその日は1日しか消費しない
- `currentDay` は選択確定時には増えず、その日の分岐エピソードを最後まで読了した時点で次日に進む
- `currentDay` は日単位、`currentEpisodeId` は分岐エピソードID単位で管理する
- `seenEpisodes` は **episodeId単位** で記録する

**例**:
- Day 2 で選択肢 A → `day-002a`
- Day 2 で選択肢 B → `day-002b`
- どちらも終了後に Day 3 へ進む

---

## R2. 複数ヒロインの条件を同時に満たした場合の優先ルール（Sec.9.1 / 9.2）

**回答**:
- 優先順位は以下で決定する
1. 個別ルート確定フラグ
2. 好感度の高いヒロイン
3. 直近3日間の接触頻度
4. 対応パラメータ達成度
5. 完全同率なら専用選択肢を表示

**詳細**:
- `route_locked_minori` のような確定フラグが立っていればそのルートを優先
- 確定前なら好感度最大のヒロインを優先
- 好感度同値なら最近よく接触している相手を優先
- それも同じなら対応パラメータが高い方を優先
- 完全同率ならプレイヤーに選ばせる専用イベントを発生させる

**MVP方針**:
- 好感度最大を優先
- 同率時は専用選択肢

---

## R3. 「最初から」の削除範囲（Sec.12.2 / 14.5）

**回答**: **進行データのみ全削除** とし、収集要素は残す。

**削除するもの**:
- オートセーブ
- 手動セーブ 1〜3枠
- 現在の進行状態
- パラメータ
- 好感度
- フラグ
- 選択履歴

**残すもの**:
- 設定データ
- ギャラリー解放状況
- 日記解放状況
- エンディング到達記録

**補足**:
- 完全初期化したい場合は別メニューで **「全データ削除」** を用意する

---

## R4. ギャラリー解放条件の正式ルール（Sec.10.2 / 12.6 / 14.3）

**回答**: ギャラリーは **画像単位で解放** する。イベント単位ではなく、CGごとに個別IDを持たせる。

**解放ルール**:
- 本編で該当CGが表示された時点で解放
- 表情差分は本編中で初表示された時点で解放
- 背景のみはギャラリー対象外
- 小物画像は将来拡張で解放対象化可能

**表示方針**:
- 未取得CGはシルエット表示あり
- 画像名は伏せる
- 解放率を表示する

**MVP対象**:
- イベントCGのみギャラリー対象
- 立ち絵差分はMVPでは対象外でも可

---

## R5. updates.json のデータ構造と掲載粒度（Sec.10.3 / 18.2 / 18.3）

**回答**: `updates.json` は **ユーザー向け更新履歴** として使う。軽微修正は載せず、公開上意味のある更新のみ記録する。

**1件として記録する対象**:
- 新規エピソード追加
- 新規画像追加
- 新規日記追加
- 新規イベント追加
- 重要な不具合修正
- 公開内容に影響する仕様変更

**原則として載せないもの**:
- 誤字修正のみ
- 内部JSON整理のみ
- 命名整理のみ
- 表示に影響しない軽微修正

**表示項目**:
- `date`
- `type`
- `title`
- `description`
- `link`（任意）

**データ構造例**:
```json
[
  {
    "date": "2026-03-10",
    "type": "episode",
    "title": "第5話を追加",
    "description": "みのりとの帰り道イベントを追加しました。",
    "link": "day-005"
  },
  {
    "date": "2026-03-11",
    "type": "fix",
    "title": "重要な進行不具合を修正",
    "description": "特定条件で次話に進めない問題を修正しました。"
  }
]
```

## R6. eventId の命名規則と episodeId との関係（Sec.10.4 / 11.4）

**回答**: `episodeId` と `eventId` は分ける。本編1話は `episodeId`、その中の個別イベントや差分は `eventId` で管理する。

**ルール**:
- `episodeId`: 1日単位または分岐1本単位
- `eventId`: その中で発生する個別イベント単位

**命名例**:
- `episodeId`: `day-005`
- `eventId`: `day-005_main`
- `eventId`: `day-005_minori_extra`
- `eventId`: `day-005_hidden_rain`

**方針**:
- `seenEpisodes` と `seenEvents` は分離
- 重複登録は許容する
- 本編メインイベントは `eventId` を持ってもよいが、保存上は別管理する

**MVP簡略化**:
- `seenEpisodes` を主使用
- `seenEvents` は将来拡張前提で構造だけ用意

## R7. 日次自動更新が失敗した場合の公開方針（Sec.3.3 / 18.1）

**回答**: 失敗時は **公開を止める**。中途半端な生成物は publish しない。

**ルール**:
- 生成途中で失敗した場合、その日の更新は公開なし
- 前日の公開状態を維持
- `updates.json` も更新しない
- 失敗ログは保存する
- 手動リカバリ後に正常公開できた日付で `updates.json` に載せる

**cron 実行順**:
1. 生成
2. 検証
3. manifest更新
4. updates更新
5. Git add
6. Git commit
7. Git push

どこかで失敗したら中断する。

## R8. BGM「2曲以上」と初期BGM一覧5曲の関係（Sec.15.3 / 24.1）

**回答**: MVPでの実装必須は **2曲**。初期BGM一覧5曲は正式版での標準セットとする。

**MVP必須**:
- `title_theme`
- `daily_theme`

**MVP任意**:
- `evening_theme`
- `tension_theme`
- `confession_theme`

**仕様整理**:
- 「初期BGM一覧5曲」は将来を含む標準セット
- MVP要件は最低2曲
- 未実装曲はプレースホルダ可
- BGM指定が未実装曲だった場合は `daily_theme` にフォールバック

**フォールバック例**:
```json
{
  "bgmFallback": "daily_theme"
}
```

## U3. 更新履歴からの「該当エピソードへリンク」の遷移先（USECASE Sec.3.3）

**回答**: 更新履歴からのリンクは、既存セーブとは独立した **閲覧モード** で開く。

**閲覧モード仕様**:
- セーブデータは更新しない
- 好感度・パラメータ・フラグも更新しない
- 既読登録も原則しない
- 将来、`previewSeen` のような別管理を追加する余地はある
- 閲覧後は更新履歴画面またはタイトルへ戻る

**リンク対象**:
- 公開済みの本編エピソードのみ対象とする
- 未到達でも閲覧モードなら開いてよい
- 分岐エピソードや隠しイベントは MVP ではリンク対象外
- MVP の `updates.json` の `link` はトップレベルの公開エピソードIDのみを対象とする

## U4. Scenario Agent の分岐先ID規則（USECASE Sec.5.2）

**回答**: `USECASE.md` の Scenario Agent プロンプトを修正し、choice の `nextEpisodeId` は **同日分岐ID** にする。

**正式ルール**:
- choice の分岐先:
  - `day-{day:03d}a`
  - `day-{day:03d}b`
- その日を読み終えた後の翌日遷移:
  - トップレベルの `nextEpisodeId`

**方針**:
- Scenario Agent に同日分岐専用テンプレートを与える
- `next_day` を choice 側で使わない
- 翌日遷移はトップレベルのみで表現する

**修正版イメージ**:

{
  "id": "day-002",
  "day": 2,
  "choices": [
    {
      "id": "c1",
      "label": "笑ってうなずく",
      "nextEpisodeId": "day-002a"
    },
    {
      "id": "c2",
      "label": "少し照れながら歩き出す",
      "nextEpisodeId": "day-002b"

## U5. 一話に登場できるヒロイン数の扱い（USECASE Sec.5.1 / 5.2）

**回答**: `characters` 配列には複数ヒロインを入れてよい。ただし MVP では **1話1メインヒロイン + サブ1人まで** に制限する。

**正式ルール**:
- `featured_heroine`: 必須 0〜1人
- `support_heroines`: 0〜1人（MVP）
- `characters` 配列には登場人物全員を入れる
- 並び順は主役を先頭にする

**MVP制限**:
- 主役ヒロイン 1人
- サポートヒロイン 最大1人
- 3人同時登場は避ける
- 主役の感情積み上げを優先する

**例**:
```json
{
  "characters": ["minori", "hinata"]
}
```

この場合、`minori` が主役、`hinata` がサポート。

## U6. Data Agent が生成する summary の用途と保存先（USECASE Sec.5.5）

**回答**: `summary` は内部メモではなく、以下の3用途に使う。

**用途1: `episodes.json` に保存**
- 各エピソードの一覧表示や管理用に使う

**用途2: Planner Agent の `recent_episodes` に渡す**
- `USECASE.md` の Planner Agent 入力にある `recent_episodes` の供給元として使う

**用途3: `updates.json` の description 生成素材**
- 新規話追加時の説明文の下書きとして流用できる

**正式保存方針**:
- `episodes.json` に `summary` を持たせる
- `updates.json` にはそのまま入れるか、短縮版を `description` に使う
- 開発ログにも使ってよい

**例**:
```json
{
  "id": "day-005",
  "day": 5,
  "title": "雨の帰り道",
  "characters": ["minori"],
  "season": "spring",
  "addedAt": "2026-03-10",
  "summary": "放課後、雨宿りをしながらみのりと少しだけ本音に近い会話を交わす。"
}
```

## F1. updates.json の更新タイミング不整合（SEQUENCE Sec.21 / USECASE Sec.4.1）

**発見**: SEQUENCE Sec.21 正常系の末尾で「Publisher Agent が GitHub Pages 反映確認後に updates.json に追記 → 再度 push」となっていた。一方 USECASE Sec.4.2 補足・QandA R7 では「episode JSON / episodes.json と同じ変更セットに含める」と確定している。

**回答**:
- **正式仕様**: updates.json は **push 前工程（Data Agent）で更新し、同一コミットに含める**
- Publisher Agent は二段 push をしない
- push 失敗時は updates.json も push されないため、公開状態は変わらない

**修正内容**:
- Seq.21: Data Agent に `updates.json の description 生成・追記` を追加
- Seq.21: Publisher Agent の「updates.json に追記 → 再度 push」を削除
- Seq.21: `git add` に `updates.json` を明示

---

## F2. オートセーブ破損時の挙動矛盾（SEQUENCE Sec.18 / USECASE Sec.3.1 / QandA U1）

**発見**: SEQUENCE Sec.18 のバージョン不一致ケースで `migrateState()` を呼び出して自動復元を試みていた。USECASE Sec.3.1 補足・QandA U1 では「自動フォールバックしない、ロード画面へ誘導する」と確定している。

**回答**:
- **正式仕様**: オートセーブの破損・バージョン不一致どちらの場合も **手動セーブへ自動フォールバックしない**
- エラーメッセージを表示し、ロード画面へ誘導する
- ユーザーが明示的にロードスロットを選ぶ

**修正内容**:
- Seq.18: バージョン不一致ケースの `migrateState()` 自動復元を削除
- Seq.18: 両ケースとも「手動セーブへ自動フォールバックしない」の Note を追加
- Seq.18: バージョン不一致ケースも `navigateTo("load")` に統一

## F3. Scenario Agent 出力と正式 episode JSON の境界不明確（SEQUENCE Sec.21）

**発見**: Seq.21 で Scenario Agent の出力が「エピソードJSON（確定）」と表記されており、これが中間生成物か正式 JSON かが不明確だった。Data Agent の変換責務も記載されていなかった。

**回答**:
- **Scenario Agent 出力**: 中間生成物（`episodeId`, `summary`, `textBlocks`, `featuredHeroine`, `supportHeroines`, `imageRequirements`, `writerComment` を含む）
- **Data Agent 出力**: 正式保存物（変換ルール適用済み）
  - `episodeId` → `id`
  - `textBlocks` → `text`
  - `featuredHeroine` + `supportHeroines` → `characters`
  - `imageRequirements` → `background` / `sceneImage` / 制作メタデータへ分解し正式 JSON に含めない
  - `writerComment` → 内部レビュー用・正式 JSON に含めない

**修正内容**:
- Seq.21: Data Agent ステップに変換責務の Note を追加
- Seq.21: HR→DA の引き渡しを「エピソードJSON（確定）」→「採用画像ファイル + 中間生成物」として明確化
