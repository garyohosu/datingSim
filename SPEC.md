## 1. 目的

本仕様書は、静的サイトとして公開可能な連載型恋愛シミュレーションゲーム
「放課後シグナル」の要件、機能、画面、データ構造、運用、音響、将来拡張を
詳細に定義する。

本作は、90年代〜2000年代初期の学園恋愛シミュレーションゲームの空気感を持つ
オリジナル作品とし、以下を成立させることを目的とする。

- ブラウザだけで遊べる
- サーバーサイド不要
- ユーザーごとの進行をブラウザ保存できる
- エピソード、画像、音楽、日記などを後から少しずつ追加できる
- AIを用いて、日次または定期的にコンテンツを増やせる
- 作品としての一貫性を保ちながら運用できる
- 本サイトは公開後の収益化を前提とし、Google AdSense の導入を必須要件とする

## 3. 公開・運用前提

### 3.1 公開基盤
- GitHub リポジトリで管理
- GitHub Pages で公開
- 生成物は静的ファイルのみ

### 3.2 実行環境
- デスクトップブラウザ優先
- スマートフォン閲覧対応
- JavaScript 有効前提
- Web Audio API 対応ブラウザ前提

### 3.3 運用方式
- 基本は「AI生成 + 人手確認」
- 日次追加を前提にするが、未更新日があってもよい
- 既存セーブを壊さないことを重視する
- 画像だけ、文章だけの追加も許容する
- 定期更新は **Windows 11 の WSL2 上で動作する OpenClaw 環境から cron により実行する**
  - 更新処理は静的コンテンツ生成、検証、manifest 更新、updates 更新、GitHub Pages 用リポジトリへのコミット・プッシュまでを含む
  - cron 想定: `0 5 * * *`（毎日早朝）
  - 失敗時はログを残し、その日の公開は行わない
  - WSL2 内で Git / Node / Python が利用可能であること
  - リポジトリパスは Linux パスで固定する
  - 途中失敗時は前日の公開状態を維持し、中途半端な生成物は push しない

### 3.4 AI開発ツール
- **Codex CLI（ChatGPT Plus 定額版）** を使用する
- シナリオ生成・JSON整形・プロンプト作成などに活用する
- 最終承認は人間が行う

## 7. ゲーム進行設計

### 7.1 基本単位
- 1日 = 1ターン
- 分岐エピソードが発生しても、その日は1日しか消費しない

### 7.2 1ターンの流れ
1. 当日の導入表示
2. 背景 / 立ち絵 / イベントCG表示
3. テキスト表示
4. 選択肢表示
5. 選択結果反映
6. 分岐先エピソードがある場合は同日内の差分エピソードへ遷移
7. 好感度・パラメータ・フラグ更新
8. オートセーブ
9. その日のエピソード読了後、次の日へ

**日次進行ルール:**
- `currentDay` は選択肢確定時には増えない
- `day-002a` / `day-002b` のような分岐エピソードは、どちらも同じ Day 2 扱いとする
- 分岐エピソードを最後まで読了した時点で次の `currentDay` に進む
- `currentDay` は日単位、`currentEpisodeId` は分岐エピソードID単位で管理する

### 7.3 プレイフェーズ
- 共通ルート
- 分岐準備期間
- 個別ルート
- 終盤イベント
- エンディング

### 7.4 スコープ計画

**総日数: 30日**

| 期間 | 内容 |
|------|------|
| 1〜10日 | 共通ルート前半 |
| 11〜20日 | 分岐準備期間（好感度で接触頻度が変化） |
| 21〜30日 | 個別ルート・終盤 |

**実装フェーズ:**
- MVP: 全体で10日分
- 拡張1: 全体で20日分
- 正式1周: 全体で30日分
- 個別ルートは30日構成の後半で差分化する

## 8. パラメータ設計

### 8.1 主人公パラメータ
- study: 学力
- sports: 運動
- charm: 魅力
- care: 気配り
- stress: ストレス

### 8.2 初期値
- 学力: 10
- 運動: 10
- 魅力: 10
- 気配り: 10
- ストレス: 0

### 8.3 上限下限
- 下限: 0
- 上限: 99

### 8.4 ヒロイン好感度
- minori（朝倉みのり）: 0〜100
- toko（白瀬透子）: 0〜100
- hinata（夏川ひなた）: 0〜100

### 8.5 内部フラグ
- 出会い済み
- 特定イベント閲覧済み
- 個別ルート侵入済み
- エンディング到達済み
- 特殊選択済み

### 8.6 ストレスの効果

stress 値が高いほど以下の影響が生じる。

| 値 | 状態 | 影響 |
|----|------|------|
| 0〜29 | 問題なし | 影響なし |
| 30〜59 | 軽い不調 | 一部選択肢の効果が弱くなる |
| 60〜79 | 会話・行動にマイナス補正 | 好感度上昇が低下 |
| 80〜99 | 要休養 | 体調不良イベント発生、失敗率上昇 |

**ストレス低下手段:**
- 休養系イベント
- 穏やかな日常イベント
- 特定ヒロインとの安らぎイベント
- 休日イベント
- 選択肢による回復

### 8.7 設計方針
- UI上は必要以上に数字を見せすぎない
- 可能なら「距離感」「雰囲気」などの抽象表現を将来導入する
- 内部管理は数値中心で行い、見た目だけ柔らかくする

## 9. ルート・分岐設計

### 9.1 ルート構造
- 序盤 1〜10日: 共通イベント中心
- 中盤 11〜20日: 好感度差で接触頻度が変化
- 後半 21日以降: 条件達成で個別ルート比率上昇

### 9.2 個別ルート条件例
- 好感度 20 以上
- 対応パラメータ 15 以上
- 必須イベント閲覧済み
- 他キャラ優先フラグなし

**複数候補が同時成立した場合の優先順位:**
1. 個別ルート確定フラグ
2. 好感度の高いヒロイン
3. 直近3日間の接触頻度
4. 対応パラメータ達成度
5. 完全同率なら専用選択肢を表示

**MVP簡略化:**
- 好感度最大を優先
- 同率時は専用選択肢を表示

### 9.3 エンディング種別
- 告白成功
- 友達エンド
- すれ違いエンド
- 通常エンド

### 9.4 分岐思想
- 一本道ではなく、選択と積み重ねで自然に寄る形
- 極端な分岐乱立は避ける
- 日次更新に強い構造にする
- 既読イベント再読時にも破綻しないようイベント単位を明確化する

## 10. コンテンツ追加モデル

### 10.1 基本追加単位
- エピソード 1 本
- 画像 1 枚

### 10.2 追加可能コンテンツ
- 本編エピソード
- 立ち絵差分
- イベントCG
- ヒロイン日記
- 隠しイベント
- 背景
- 季節イベント
- ボーナス短文

### 10.3 更新方針
- 毎日追加を理想とする
- ただし「画像のみ」「日記のみ」「短編イベントのみ」でも可
- 更新履歴をトップページに表示する
- `updates.json` は **ユーザー向け更新履歴** とし、公開上意味のある更新のみ記録する
- 誤字修正のみ、内部JSON整理のみ、命名整理のみ、表示に影響しない軽微修正は原則掲載しない
- 記録対象は、新規エピソード追加、新規画像追加、新規日記追加、新規イベント追加、重要な不具合修正、公開内容に影響する仕様変更とする
- 更新履歴からのリンクは、通常進行とは独立した **閲覧モード** で開く
- 閲覧モードではセーブ、好感度、パラメータ、フラグ、既読状態を更新しない
- MVPでは公開済みの本編エピソードのみを閲覧モードのリンク対象とし、分岐エピソードや隠しイベントは対象外とする

### 10.4 過去日の増補
- 条件達成により、過去の日付に隠しイベントが追加される仕様を将来導入可能
- 過去イベント追加時も既存セーブが壊れないよう **eventId ベースで管理する**
- `episodeId` は本編1話または分岐1本単位、`eventId` はその中の個別イベント単位として分離する
- `seenEpisodes` は episodeId 単位、`seenEvents` は個別イベント単位で分けて管理する
- 本編メインイベントは `eventId` を持ってもよいが、保存上は `seenEpisodes` と `seenEvents` を別管理する
- 過去追加イベントは既存セーブでも未読扱いで再生可能とし、一覧に「新規追加」として表示する
- MVPでは `seenEpisodes` を主使用し、`seenEvents` は将来拡張前提で構造のみ先に用意する

### 12.2 タイトル画面

表示項目:
- タイトルロゴ
- 続きから
- 最初から
- ロード
- 設定
- 更新情報
- ギャラリー

動作:
- セーブあり: 続きからを強調
- セーブなし: 最初からを強調
- `続きから` は **オートセーブ専用** とする
- オートセーブ破損時は自動で手動セーブに切り替えず、エラーメッセージを表示して `ロード` 画面へ誘導する
- `ロード` 画面では `オートセーブ / 手動1 / 手動2 / 手動3` を一覧表示する
- `最初から` は進行データのみ削除して初期状態へ遷移する
- 設定データ、ギャラリー解放、日記解放、エンディング到達記録は保持する
- 完全初期化が必要な場合は別メニューの「全データ削除」で対応する

### 12.3 メインゲーム画面

表示項目:
- 背景
- キャラ立ち絵またはイベントCG
- テキストウィンドウ
- キャラ名欄
- 選択肢
- スキップ/ログ/設定ボタン

### 16.1 画像の方向性

- 明るい制服
- 柔らかいアニメ塗り

### 16.2 画像種別
- 背景
- 立ち絵
- 表情差分
- イベントCG
- タイトル画像
- 小物画像

### 16.3 初期必須画像（MVP最低枚数）

**背景 5枚（必須）:**
1. タイトル背景
2. 教室背景
3. 校門背景
4. 帰り道背景
5. 図書室背景

**キャラクター素材:**
- ヒロイン3人の基本立ち絵
- 各ヒロイン3表情

### 16.4 推奨命名
- `minori_default.webp`
- `minori_smile.webp`
- `bg_school_gate_morning.webp`
- `event_day001.webp`

### 16.5 画像制作運用方針
- 初期キャラクター立ち絵、表情差分、タイトル用キービジュアル、重要イベントCGは手動生成を正本とする。
- 初期キャラクター画像の制作には **Genspark AI Designer** および **Nano Banana 2** を利用可能とする。

### 16.6 画像生成運用フロー
1. Image Prompt Agent がプロンプト作成
2. Genspark / DALL-E 等で人間が画像生成を実行
3. Consistency Agent が一貫性チェックを補助
4. **最終採用判断は人間が行う**（キャラの顔ブレは致命的なため）

## 17. シナリオ仕様

### 17.1 文体方針
- 読みやすい
- くどすぎない
- 感情の積み上がりを重視
- セリフと地の文のバランスを取る

### 17.2 禁止事項
- キャラ性崩壊
- 急な過激展開
- 一話ごとの温度差が大きすぎること
- 設定矛盾
- 学園恋愛ゲームの空気感を壊すメタ表現

### 17.3 1話のボリューム目安
- **「1回」= 1エピソード（= 1日分）を指す**
- 導入 + 本文 + 選択肢で 400〜1200文字程度
- 短い日も許容
- 特別イベント日は長めでも可

### 17.4 日記仕様
- ヒロイン視点
- 本編補完
- 100〜400文字程度
- **解放条件: 対応する本編エピソードの読了**

## 18. 更新フロー

### 18.1 更新手順
1. 次話の企画を作成
2. シナリオ草案生成（Codex CLI 使用）
3. 画像プロンプト生成
4. 画像生成・採用判断
5. 整合性確認
6. episode JSON 化
7. manifest 更新
8. updates 更新
9. GitHub へ反映（cron / 手動）
10. 公開確認

**公開ルール:**
- 生成、検証、manifest更新、updates更新、Git add、Git commit、Git push の全工程が成功した場合のみ公開する
- 途中で失敗した場合は処理を中断し、前日の公開状態を維持する

### 18.2 更新対象ファイル
- `assets/data/episodes/day-xxx.json`
- `assets/data/manifests/episodes.json`
- `assets/data/manifests/updates.json`
- `assets/images/events/...`
- 必要に応じて `gallery.json` や `diaries/`

`episodes.json` には、一覧管理・Planner Agent入力・更新履歴文生成素材として使うため、短い `summary` を保持してよい。

### 18.3 更新ポリシー
- 既存 ID を変更しない
- 公開済みファイル名をなるべく変えない
- manifest に追加して順序を管理する
- **誤字・バグ・JSON不整合は修正可**。ただし以下を守る:
  - ID は原則変更しない
  - セーブ互換性を壊す変更は避ける
  - 大きな内容変更は新規イベントとして追加する
  - 修正履歴は `updates.json` に記録する
- `updates.json` の表示項目は `date`, `type`, `title`, `description`, `link`（任意）を基本とする

## 28. 開発メモ

- 最初から大作を目指さず、10日分の完成度を高める
- 画像・音・文章を少しずつ育てる
- 「量産」より「雰囲気の一貫性」を重視する
- 恋愛ゲームはフラグ管理で壊れやすいので、IDと状態遷移を丁寧に扱う
- 毎日追加型は魅力的だが、運用負荷を軽くするために更新粒度を柔軟にする
